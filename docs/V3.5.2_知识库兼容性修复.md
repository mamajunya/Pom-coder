# V3.5.2 知识库兼容性修复

## 📋 问题描述

用户报告构建知识库时出现错误：
```
✗ 错误: 'code'
```

**根本原因**：V3.5代码切片器生成的JSON格式与`build_knowledge_base_npu.py`期望的格式不匹配。

## 🔍 格式差异

### 旧格式（V3.0之前）
```json
{
  "code": "...",
  "path": "...",
  "language": "...",
  "description": "...",
  "metadata": {}
}
```

### V3.5新格式
```json
{
  "id": "abc123...",
  "content": "...",           // ❌ 不是"code"
  "file_path": "...",         // ❌ 不是"path"
  "language": "...",
  "tokens": 150,
  "start_line": 42,
  "end_line": 45,
  "line_indices": [42, 43, 44, 45],
  "chunk_type": "function",
  "name": "my_function",
  "summary": "...",           // ✅ 新增
  "imports": [...],
  "context": {...},           // ✅ 新增
  "metadata": {...},
  "quality_score": 9.0,       // ✅ 新增
  "semantic_completeness": 1.0 // ✅ 新增
}
```

## ✅ 修复方案

修改`build_knowledge_base_npu.py`的两个方法，使其兼容新旧两种格式：

### 1. `collect_from_json` 方法

**修改位置**：第157-169行

**核心改动**：
```python
# ✅ 兼容V3.5格式：content/code, file_path/path
code = item.get('content', item.get('code', ''))
path = item.get('file_path', item.get('path', 'unknown'))

# 生成描述（优先使用summary）
description = item.get('summary', item.get('description', ''))
if not description and item.get('name'):
    chunk_type = item.get('chunk_type', 'code')
    description = f"{chunk_type}: {item['name']}"

# ✅ 保留V3.5特有字段到metadata
metadata = item.get('metadata', {})
if 'tokens' in item:
    metadata['tokens'] = item['tokens']
if 'quality_score' in item:
    metadata['quality_score'] = item['quality_score']
if 'semantic_completeness' in item:
    metadata['semantic_completeness'] = item['semantic_completeness']
# ... 更多字段
```

### 2. `load_existing_knowledge_base` 方法

**修改位置**：第182-195行

**核心改动**：
```python
# ✅ 兼容V3.5格式
code = item.get('content', item.get('code', ''))
path = item.get('file_path', item.get('path', 'unknown'))

snippet = CodeSnippet(
    code=code,
    path=path,
    language=item.get('language', 'unknown'),
    description=item.get('description', item.get('summary', '')),
    metadata=item.get('metadata', {})
)
```

## 🎯 兼容性保证

修复后的代码能够：

✅ **向后兼容**：仍然支持旧格式（`code`、`path`字段）  
✅ **向前兼容**：完全支持V3.5新格式（`content`、`file_path`字段）  
✅ **保留增强信息**：V3.5的质量评分、语义完整性等字段会保存到metadata  
✅ **智能描述生成**：优先使用`summary`，否则从`name`和`chunk_type`生成

## 📊 V3.5新字段的利用

修复后，V3.5的增强字段会被保留在`metadata`中，可用于：

- **质量过滤**：根据`quality_score`筛选高质量代码块
- **语义完整性**：使用`semantic_completeness`判断代码块是否完整
- **精确定位**：通过`start_line`、`end_line`、`line_indices`精确定位
- **上下文增强**：利用`context`字段提供更丰富的检索上下文
- **Token管理**：使用`tokens`字段优化批处理大小

如果你已经有旧格式的知识库：

1. **无需重建**：修复后的代码能直接读取旧格式
2. **增量更新**：使用`--append-mode`叠加新的V3.5格式数据
3. **完全重建**：如果想利用V3.5的增强特性，建议重新切片并重建知识库

## ⚠️ 注意事项

1. **字段优先级**：
   - 代码内容：`content` > `code`
   - 文件路径：`file_path` > `path`
   - 描述信息：`summary` > `description` > 自动生成

2. **metadata扩展**：V3.5的额外字段会自动添加到`metadata`，不会丢失

3. **性能影响**：兼容性检查对性能影响可忽略（每个item只增加几次字典查询）
